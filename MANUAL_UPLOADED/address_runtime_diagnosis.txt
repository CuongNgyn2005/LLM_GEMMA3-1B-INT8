Address/runtime diagnosis for ZCU104 FPGA path
Date: 2026-06-06

1. Address check

fpga_host.cpp uses:
- VPU_BASE_PHYS      = 0xA0000000
- VPU_MMAP_SIZE      = 0x00300000
- ACT_BASE local     = 0x00010000
- WEIGHT_BASE local  = 0x00100000
- RESULT_BASE local  = 0x00200000

Therefore the host touches physical addresses:
- Registers: 0xA0000000 - 0xA0000080
- ACT:       starts at 0xA0010000
- WEIGHT:    starts at 0xA0100000
- RESULT:    starts at 0xA0200000
- mmap end:  0xA02FFFFF

RTL/Vivado exposes:
- IP segment: 0xA0000000 - 0xAFFFFFFF
- AXI4_Mapping VPU_BASE_ADDR = 40'h00A0_0000_00
- ACT window local    0x00010000 - 0x0001FFFF
- WEIGHT window local 0x00100000 - 0x001FFFFF
- RESULT window local 0x00200000 - 0x0020FFFF

Conclusion:
- The host does not exceed the IP address range from promp.md.
- The current host does not mmap or write the DDR range 0x70000000 - 0x7FFFFFFF.
- The DDR range is documented in logs only because the current RTL path uses IP-local BRAM windows instead of a DMA/DDR pointer interface.

3. Highest-risk remaining hardware question

The RTL testbench writes full 128-bit AXI beats with wstrb=16'hffff.
The Linux host writes the ACT/WEIGHT 16-byte beat as four 32-bit MMIO stores.

This can work if the PS/interconnect emits correct WDATA/WSTRB byte lanes for narrow stores.
However the Vivado design_1.bd contains SUPPORTS_NARROW_BURST=0 on the SmartConnect path, so this must be tested on real hardware.

The standalone diagnostic DATN_RTL/MANUAL_UPLOADED/fpga_mmio_smoke_test.cpp tests exactly this path:
- write 32 activation int8 values of 1
- write 32 weight int8 values of 1
- start one-row VPU computation
- expect result 32

If this returns anything other than 32 or times out, the issue is in PS-to-PL MMIO/write-strobe behavior or the loaded bitstream/interconnect, not in the llama model.
